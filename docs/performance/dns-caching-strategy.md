# DNS Caching Strategy

This document defines the caching strategy for PowerBlockade, including baseline cache settings, image versions, precache policies, domain selection criteria, and primary/secondary synchronization.

## Precache Policy

The precache system proactively warms DNS cache entries for high-frequency domains to reduce latency on first query after cache expiry or service restart.

### Domain Selection Criteria

Domains are selected for precaching based on the following criteria:

1. **Query Frequency**: Only domains with successful queries in the last 24 hours are candidates
2. **Success Requirement**: Queries must have returned `rcode=0` (NOERROR)
3. **Non-Blocked Status**: Blocked domains (`blocked=False`) are explicitly excluded
4. **Ranking**: Domains are ranked by query count, with the highest-frequency domains prioritized

**Selection SQL Logic** (from `admin-ui/app/services/precache.py`):

```sql
SELECT qname, COUNT(id) as count
FROM dns_query_events
WHERE ts >= (NOW() - INTERVAL '24 hours')
  AND blocked = FALSE
  AND rcode = 0
GROUP BY qname
ORDER BY count DESC
LIMIT :domain_count
```

### Domain Exclusion Criteria

The following domain categories should **never** be precached due to volatility or security concerns:

| Category | Rationale | Examples |
|----------|-----------|----------|
| **CDN/Edge Domains** | IP addresses change frequently based on geography and load | `*.cloudfront.net`, `*.cdn.cloudflare.com`, `*.akamaized.net` |
| **Security-Sensitive Domains** | Stale responses could create security risks | Banking, authentication, certificate revocation |
| **Dynamic DNS Services** | TTL typically very short, IPs change frequently | `*.dyndns.org`, `*.no-ip.com`, `*.ddns.net` |
| **Short-TTL Domains** | Domains with TTL < 60 seconds should be refreshed at query time | Some API endpoints, load balancer VIPs |
| **Blocked Domains** | Already filtered out by query logic | All domains matching RPZ blocklists |

**Recommended Exclusion Patterns** (for future implementation):

```
# CDN patterns
*.cloudfront.net
*.akamaized.net
*.fastly.net
*.cdn.cloudflare.com

# Dynamic DNS
*.dyndns.*
*.no-ip.*

# Security-sensitive
*.bank.*
*.paypal.*

```

### Refresh Cadence Policy

The precache system supports two refresh modes:

#### TTL-Aware Refresh (Default: `precache_ignore_ttl=false`)

- **Behavior**: Respects upstream TTL with a 20% safety margin
- **Formula**: `refresh_threshold = ttl - max(ttl * 0.2, 30 seconds)`
- **Example**: A domain with 300s TTL refreshes at 240s (300 - 60)
- **Minimum Margin**: 30 seconds (prevents rapid refresh of short-TTL domains)

#### Fixed Interval Refresh (Override: `precache_ignore_ttl=true`)

- **Behavior**: Ignores upstream TTL, refreshes on fixed schedule
- **Default**: 60 minutes (`precache_custom_refresh_minutes`)
- **Use Case**: Environments with unstable upstream DNS where TTL is unreliable

#### Primary Node Schedule

| Setting | Default | Description |
|---------|---------|-------------|
| `precache_refresh_minutes` | 30 | How often the precache job runs (checks which domains need warming) |
| `precache_custom_refresh_minutes` | 60 | Fixed refresh interval when TTL is ignored |

### TTL-Respect Policy

| Policy | Setting | Behavior |
|--------|---------|----------|
| **Honor TTL** | `precache_ignore_ttl=false` (default) | Domains refreshed before TTL expires with 20% safety margin |
| **Ignore TTL** | `precache_ignore_ttl=true` | All domains refreshed on fixed interval regardless of upstream TTL |

**Recommendation**: Use TTL-aware refresh (default) for most deployments. Only use fixed interval if upstream DNS servers provide unreliable TTL values.

### Primary vs Secondary Sync Timing

#### Secondary Node Precache Behavior

Secondary nodes run their own precache warming independently but use the **primary node's domain list** as the source of truth.

**Sync Flow**:

1. Primary node computes top domains from its query log (24h window, top N domains)
2. Secondary fetches domain list via `/api/node-sync/precache-domains`
3. Secondary resolves each domain against its **local recursor** (not primary)
4. Secondary warms its local cache with resolved entries

**Timing Configuration** (sync-agent):

| Setting | Default | Description |
|---------|---------|-------------|
| `PRECACHE_INTERVAL_SECONDS` | 300 | Seconds between secondary precache runs |
| `CONFIG_SYNC_INTERVAL_SECONDS` | 300 | Seconds between config sync checks |

#### Expected Drift Windows

| Scenario | Drift | Impact |
|----------|-------|--------|
| Normal operation | 0-5 minutes | Negligible - both nodes warm similar domains |
| Primary outage | Up to `PRECACHE_INTERVAL_SECONDS` | Secondary continues using last-known-good domain list |
| New high-traffic domain | Up to 24 hours | Domain appears in next day's top domains list |

**Drift Tolerance**: Secondaries may have slightly different cache contents than primary, but all nodes resolve from authoritative upstream, so results remain consistent.

### Risk Assessment

#### Stale Cache Risks

| Risk | Mitigation |
|------|------------|
| IP address changes | TTL-aware refresh with safety margin |
| CDN routing changes | Exclude CDN domains from precache |
| DNS failover events | Short TTL domains refresh more frequently |

#### Premature Refresh Risks

| Risk | Mitigation |
|------|------------|
| Increased upstream queries | Safety margin prevents excessive refresh |
| Cache thrashing | Minimum 30s margin on short-TTL domains |
| Resource exhaustion | Batch processing with 100ms delay between batches |

| `precache_ignore_ttl` | false | boolean | Ignore upstream TTL |
| `precache_custom_refresh_minutes` | 60 | 10-1440 | Fixed refresh interval when TTL ignored |
| `precache_dns_server` | recursor | hostname | DNS server to query for warming |
| `precache_dns_port` | 5300 | 1-65535 | Port for DNS queries |

### Implementation Details

**Primary Node** (`admin-ui/app/services/precache.py`):
- `get_top_domains_to_warm()`: Queries last 24h of successful, non-blocked queries
- `get_domains_needing_refresh()`: Filters domains that are stale based on TTL/interval
- `warm_cache()`: Batch resolves domains with 50/batch, 100ms inter-batch delay

**Secondary Node** (`sync-agent/agent.py`):
- `fetch_precache_domains()`: Retrieves domain list from primary
- `run_precache_warming()`: Resolves domains against local recursor
- Runs every `PRECACHE_INTERVAL_SECONDS` (default: 5 minutes)

### Monitoring and Observability

**Metrics Available**:
- `cached_domains`: Total domains in TTL cache
- `fresh`: Domains still within TTL
- `expired`: Domains past TTL needing refresh

**Log Indicators**:
- `Precache warming: N/M domains in Xms` - Successful warming
- `All N domains still fresh, skipping` - No refresh needed
- `Failed to warm domain: error` - Individual resolution failure

### Best Practices

1. **Start conservative**: Begin with 1000-2000 domains, increase based on memory
2. **Monitor cache hit rate**: If hit rate drops, may need more domains
3. **Exclude volatile domains**: Implement exclusion patterns for CDNs and dynamic DNS
4. **Tune refresh interval**: Balance between freshness and upstream query load
5. **Secondary autonomy**: Each secondary warms independently for resilience

# Tuning Matrix

This section provides a setting-by-setting matrix for cache and precache tuning, including safe ranges, expected effects, failure modes, and rollback actions.

## One-Layer-at-a-Time Guidance

**CRITICAL**: Never tune dnsdist AND recursor cache settings simultaneously. Each layer must be validated independently before moving to the next layer.

**Recommended tuning sequence**:
1. **Phase 1**: dnsdist packet cache only
2. **Phase 2**: Recursor cache only (after dnsdist validated)
3. **Phase 3**: Precache scheduler knobs only (after both cache layers validated)

**Why this matters**: Tuning multiple layers simultaneously makes it impossible to attribute performance changes or issues to a specific setting. If something breaks, you won't know which layer caused it.

## dnsdist Packet Cache Tuning Matrix

These settings control the first-line packet cache at the dnsdist layer. Queries that hit this cache never reach the recursor.

| Setting | Current Value | Safe Range | Expected Effect | Failure Mode | Rollback Action |
|---------|---------------|------------|-----------------|--------------|-----------------|
| `maxEntries` | 500000 | 100000-1000000 | More entries = higher hit rate for diverse query sets | OOM if set too high for available RAM; pre-allocation wastes memory if set too high | Reduce by 50% and restart dnsdist container |
| `maxTTL` | 86400 (24h) | 3600-172800 | Longer TTL = more cache hits but potentially stale data | Stale DNS responses if upstream IPs change; blocked domain updates delayed | Reduce to 3600 and flush cache with `getPool(''):getCache():expunge(0)` |
| `minTTL` | 1 | 0-60 | Filters out very short-TTL entries from cache | Entries with TTL < minTTL bypass cache, increasing upstream load | Set to 0 or 1 |
| `temporaryFailureTTL` | 5 | 0-60 | Caches SERVFAIL/REFUSED responses briefly | 0 = no caching of failures = more upstream load; too high = clients see cached failures | Reduce to 0 to disable failure caching |
| `staleTTL` | 60 | 0-300 | TTL used when serving stale entries during backend outage | 0 = no stale serving = errors during outages; too high = very stale data | Reduce to 0 or lower to 30 |
| `dontAge` | false | true/false | true = entries keep original TTL; false = TTL counts down | true = stale data appears fresh to clients; false = normal behavior | Set to false |
| `shuffle` | true | true/false | true = randomize RR order (load distribution); false = preserve order | May affect clients expecting consistent ordering | Set to false |
| `setStaleCacheEntriesTTL()` | 60 | 0-300 | Max age of expired entries eligible for stale serving | 0 = stale disabled; too high = very old data served | Reduce to 0 or 30 |

### dnsdist Tuning Notes

- **Memory impact**: Each entry pre-allocates ~12-20 bytes plus response size. 500k entries with 512-byte avg = ~250 MB.
- **Cache collision warning**: Very large caches may have more hash collisions. Monitor via `getPool(''):getCache():printStats()`.
- **Stale serving interaction**: Both `staleTTL` (in cache config) and `setStaleCacheEntriesTTL()` affect stale behavior. Keep them consistent.

## Recursor Cache Tuning Matrix

These settings control the recursor's record and packet caches. The recursor handles recursive resolution and has two cache layers.

### Record Cache Settings

| Setting | Current Value | Safe Range | Expected Effect | Failure Mode | Rollback Action |
|---------|---------------|------------|-----------------|--------------|-----------------|
| `max-cache-entries` | 2000000 | 500000-5000000 | More entries = more cached records for resolution | OOM; excessive memory pressure on container | Reduce by 50% and restart recursor |
| `max-cache-ttl` | (default) | 86400-604800 | Max time records stay in cache | Stale NS/A records cause resolution failures | Not currently set; add with conservative value |
| `max-negative-ttl` | (default) | 60-3600 | Max time NXDOMAIN/NODATA cached | Too high = blocked domains appear blocked after unblock; too low = extra upstream queries | Not currently set; add with 300s value |

### Packet Cache Settings

| Setting | Current Value | Safe Range | Expected Effect | Failure Mode | Rollback Action |
|---------|---------------|------------|-----------------|--------------|-----------------|
| `max-packetcache-entries` | 1000000 | 100000-2000000 | More entries = more cached response packets | OOM; memory pressure | Reduce by 50% and restart |
| `packetcache-ttl` | 86400 | 3600-172800 | Max TTL for cached response packets | Stale responses; delayed blocklist updates | Reduce to 3600, flush with `rec_control wipe-cache '$'` |
| `packetcache-negative-ttl` | 60 | 0-300 | TTL for NXDOMAIN/NODATA responses | 0 = no negative caching; too high = stale NXDOMAIN | Reduce to 30 or 0 |
| `packetcache-servfail-ttl` | 5 | 0-60 | TTL for SERVFAIL responses | 0 = no caching; too high = clients see cached failures | Reduce to 0 |
| `disable-packetcache` | no | yes/no | Disables packet cache entirely | yes = all queries hit record cache = slower | Ensure set to 'no' |
| `packetcache-shards` | (default) | 1-1024 | Number of hash shards for packet cache | Too few = lock contention; too many = memory overhead | Not currently set; tune only if needed |

### Recursor Tuning Notes

- **Two cache types**: Record cache stores individual DNS records; packet cache stores complete response packets. Both matter.
- **Memory math**: 2M record entries @ 500 bytes avg = ~1 GB. 1M packet entries @ 300 bytes avg = ~300 MB.
- **Negative cache interaction**: `packetcache-negative-ttl` affects how quickly unblocks propagate. Keep low (60s max).
- **Flush command**: `rec_control wipe-cache '$'` clears all caches without restart.

These knobs control the proactive cache warming system that pre-populates cache entries for high-frequency domains.

| Setting | Current Value | Safe Range | Expected Effect | Failure Mode | Rollback Action |
|---------|---------------|------------|-----------------|--------------|-----------------|

### Secondary Node Precache Knobs

| Setting | Current Value | Safe Range | Expected Effect | Failure Mode | Rollback Action |
|---------|---------------|------------|-----------------|--------------|-----------------|
| `PRECACHE_INTERVAL_SECONDS` | 300 | 60-3600 | Seconds between secondary precache runs | Too low = excessive CPU/network; too high = stale cache | Increase to 600 if concerned |

### Precache Tuning Notes

- **TTL safety margin**: System uses `max(ttl * 0.2, 30)` as refresh margin. Short TTL domains (< 150s) always use 30s margin.
- **Batch processing**: Domains warmed in batches of 50 with 100ms delay between batches. This is hardcoded.
- **Secondary autonomy**: Secondaries fetch domain LIST from primary but resolve locally. Primary outage = secondaries use stale list.
- **Exclusion gap**: Current implementation lacks CDN/dynamic DNS exclusion patterns. This should be addressed.

## Interaction Warnings

### Cache Size vs Memory Usage

| Interaction | Warning |
|-------------|---------|
| dnsdist `maxEntries` + recursor caches | Total memory = dnsdist packet cache + recursor packet cache + recursor record cache. Monitor container memory before increasing any cache size. |
| `max-cache-entries` + `max-packetcache-entries` | Both contribute to recursor memory. Doubling both = ~2x recursor memory. |
| High `precache_domain_count` + large caches | Precache warming fills caches. If cache sizes are small relative to domain count, warming provides limited benefit. |

### TTL Settings vs Stale Data

| Interaction | Warning |
|-------------|---------|
| `maxTTL` + blocklist updates | Blocklist changes may not take effect until cache entries expire. High maxTTL = delayed blocking. |
| `packetcache-negative-ttl` + unblocking | Domains removed from blocklist may still return NXDOMAIN until negative cache expires. Keep this value low. |
| `dontAge=true` + any TTL | Entries never appear to age from client perspective. Can mask stale data. Avoid unless specifically needed. |
| `precache_ignore_ttl=true` + short-TTL domains | Short-TTL domains will be cached longer than intended by domain owner. May cause issues with CDNs/load balancers. |

### Negative Cache vs Blocking Propagation

| Interaction | Warning |
|-------------|---------|
| `packetcache-negative-ttl` + RPZ blocking | Blocked domains return NXDOMAIN. If negative TTL is high and domain is unblocked, NXDOMAIN persists. |
| dnsdist `temporaryFailureTTL` + upstream issues | Cached SERVFAIL responses mask transient upstream issues. Clients may see failures even after upstream recovers. |
| `max-negative-ttl` + new domains | Newly created domains may be cached as NXDOMAIN if queried before delegation. |

### Stale Serving Interactions

| Interaction | Warning |
|-------------|---------|
| `staleTTL` + `setStaleCacheEntriesTTL()` | Both must be set for stale serving to work during outages. If either is 0, stale serving is effectively disabled. |
| Stale serving + blocklist changes | Stale responses may include records for domains that were just blocked. Acceptable tradeoff for availability. |
| Stale serving + DNSSEC | Stale responses may have expired RRSIG records. DNSSEC-validating clients may reject them. |

## Tuning Change Procedure

### Before Any Change

1. **Record baseline**: Note current cache hit rate, latency, memory usage
2. **Check available memory**: Ensure container has headroom for cache growth
3. **Plan rollback**: Know exact command to revert change

### dnsdist Changes

```bash
# 1. Edit config
vim dnsdist/dnsdist.conf.template

# 2. Validate syntax (dnsdist --check-config)
docker compose exec dnsdist dnsdist --check-config -C /etc/dnsdist/dnsdist.conf

# 3. Restart dnsdist only
docker compose restart dnsdist

# 4. Verify cache is working
docker compose exec dnsdist dnsdist -e "getPool(''):getCache():printStats()"

# Rollback if needed
git checkout -- dnsdist/dnsdist.conf.template
docker compose restart dnsdist
```

### Recursor Changes

```bash
# 1. Edit config
vim recursor/recursor.conf.template

# 2. Validate config
# (Recursor validates on startup; check logs for errors)

# 3. Restart recursor only
docker compose restart recursor

# 4. Verify cache is working
docker compose exec recursor rec_control get all-outqueries

# Alternative: Flush cache without restart
# docker compose exec recursor rec_control wipe-cache '$'

# Rollback if needed
git checkout -- recursor/recursor.conf.template
docker compose restart recursor
```
