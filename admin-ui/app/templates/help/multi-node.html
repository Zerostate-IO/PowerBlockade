Multi-Node Architecture

Deploying multiple DNS servers with centralized logging and configuration.

Why multiple nodes?

Running multiple DNS nodes provides:

  • High availability – One node down? Others keep serving DNS
  • Reduced latency – Place nodes closer to clients (home/office/remote sites)
  • Resilience – Local DNS continues working during network partitions (with planned buffering)
  • Scalability – Handle more queries by distributing load

Architecture overview

PowerBlockade uses a hub-and-spoke model:

  • Primary node runs the Admin UI API, Postgres, dnstap-processor, and the full monitoring stack
  • Secondary nodes are lightweight: just PowerDNS Recursor + sync-agent
  • All nodes send DNS query events to the primary via POST /api/node-sync/ingest
  • Secondaries pull RPZ/forward-zone config from primary via sync-agent

This design provides a single pane of glass for centralized logging and config management, without duplicating the heavy UI/database infrastructure on every node.

Node registration and heartbeats

Secondary nodes register with the primary using an API key:

  1. Generate a secondary node on the primary (creates api_key)
  2. Deploy sync-agent on the secondary with that key
  3. sync-agent sends heartbeats every 60 seconds (configurable via HEARTBEAT_INTERVAL_SECONDS)
  4. If primary misses ~5 consecutive heartbeats (5 min), the node shows as offline

💡 Planned: sync-agent will buffer query events locally during network partitions and retry upload when connectivity returns, with idempotent deduplication (via event_id).

Config synchronization

Blocklists and forward zones defined on the primary are synchronized to all secondary nodes:

  • Primary generates RPZ zone files → shared volume → recursor-reloader reloads on trigger
  • sync-agent on secondaries pulls latest RPZ files and forward zones from primary
  • Secondary recursor reloads immediately when files change (trigger-driven)

✅ Config sync is fully implemented. Blocklists, forward zones, and whitelist/blacklist entries sync automatically.

Network topology

Typical deployment: 

                   Primary Node
             (UI + API + Postgres + dnstap-processor)
                        ↓
                   dnstap ingestion
                        ↓
    ┌──────────────────┴──────────────────┐
    │                                     │
Secondary 1                       Secondary 2
  (home network)                    (office network)
  192.168.1.10                      10.0.0.10
    DNS:53                           DNS:53

Each site's DHCP points clients to its local secondary for low-latency DNS, while all query data centralizes at the primary for unified visibility.

Troubleshooting

Permission denied errors

If sync-agent logs show Permission denied: forward-zones.conf: 

# Restart the stack to re-run init-permissions
docker compose --profile sync-agent down
docker compose --profile sync-agent up -d

The init-permissions service automatically fixes file ownership on startup.

Node shows offline but sync-agent is running

Check connectivity to primary: 

# On secondary, test connection to primary
curl -I http://PRIMARY_IP:8080/health

# Check sync-agent logs
docker compose logs sync-agent --tail=50

Config not syncing

Verify environment variables are correct on the secondary: 

# Check .env has these set correctly
NODE_NAME=your-secondary-name
PRIMARY_URL=http://PRIMARY_IP:8080
PRIMARY_API_KEY=your-key-from-nodes-page
